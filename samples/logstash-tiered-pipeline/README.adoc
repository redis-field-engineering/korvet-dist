= Logstash Tiered Pipeline
:toc:
:icons: font

== Overview

This sample runs a local end-to-end Korvet pipeline with Docker Compose:

* Redis Enterprise DB on `localhost:12000` as the hot tier backend
* `minio/minio` as an S3-compatible cold storage backend
* `redisfield/korvet:early-access` with tiered storage enabled against S3
* One Logstash producer container with ten pipelines, each generating synthetic log events into one of `logs-01` through `logs-10`
* One Logstash consumer container with ten pipelines, each reading one source topic, enriching events, and writing to `logs-processed`

Before Korvet starts, `make up` resets the S3 backend to an empty state by removing local MinIO data and flushes the Redis Enterprise DB on `localhost:12000` for a clean run.

The 10 source topics are created or updated with these Korvet settings:

* `bucket.duration.ms=60000` (1 minute)
* `retention.ms=86400000` (1 day)
* `local.retention.bytes=214748364` (80% of the default 256 MiB Redis budget)
* `remote.storage.enable=true`
* `partitions=4`

The `logs-processed` topic is also created or updated with:

* `local.retention.bytes=214748364`

== Quick Start

[source,bash]
----
cd samples/logstash-tiered-pipeline
make up
make describe-topics
make consume
----

`make up` starts MinIO, Korvet, and Kafka tools first, waits until Korvet accepts Kafka admin requests, ensures the topics exist with the required settings, and only then starts the Logstash producer and consumer containers, each with ten topic-specific pipelines.
It also starts MinIO on `localhost:9000` with the console on `localhost:9001`.

== Services

* `korvet` exposes Kafka on `localhost:9092`
* `korvet` health is available on http://localhost:8080/actuator/health
* MinIO exposes S3-compatible storage on http://localhost:9000
* MinIO console is available at http://localhost:9001 (`minioadmin` / `minioadmin`)
* sample data is reset under `samples/logstash-tiered-pipeline/data/` on each `make up`

== Notes

This sample uses an S3-compatible URI for Korvet remote storage:

[source,text]
----
s3a://korvet/delta
----

Korvet writes archived topic data to MinIO using static credentials configured in `docker-compose.yml`.

== Korvet Tuning

With the current `redisfield/korvet:early-access` image, the Logstash consumer can stall under the full 40-partition subscription unless Korvet is given more headroom for Redis connections, server backpressure, and archival cadence.

When that happens, the usual symptoms are:

* `logs-processed` stays at offset `0`
* `kafka-console-consumer` on `logs-processed` times out
* one or more Logstash consumers log heartbeat or coordinator timeouts

For a stable run, apply these Korvet environment overrides in `docker-compose.yml`:

[source,yaml]
----
KORVET_REDIS_POOL_SIZE: 64
KORVET_REDIS_METADATA_POOL_SIZE: 64
KORVET_REDIS_ARCHIVAL_SOURCE_POOL_SIZE: 64
KORVET_REDIS_COMMITTED_OFFSET_POOL_SIZE: 64
KORVET_REDIS_POOL_MAX_WAIT: 10s
KORVET_SERVER_MAX_PENDING_BYTES: 268435456
KORVET_SERVER_RESUME_PENDING_BYTES: 134217728
KORVET_STORAGE_REMOTE_ARCHIVER_INTERVAL: 60s
KORVET_STORAGE_REMOTE_ARCHIVER_BATCH_SIZE: 100000
----

These settings were validated against the current sample wiring:

* Redis Enterprise DB on `localhost:12000`
* MinIO-backed cold storage
* one Logstash consumer container running ten pipelines, one per source topic
