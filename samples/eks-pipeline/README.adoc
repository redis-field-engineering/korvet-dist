= EKS Data Pipeline Sample
:toc:
:icons: font

Complete data pipeline on AWS EKS with Logstash, Korvet, Redis Enterprise, and tiered storage to S3.

== Components

[cols="1,2,2"]
|===
|Component |Configuration |Purpose

|Redis Enterprise
|3 nodes, 4 shards, OSS Cluster API
|Hot storage tier

|Korvet
|Single broker, tiered storage enabled
|Kafka-compatible API, archival to S3

|Logstash
|Generator input, Kafka output
|Data generation for testing

|Spark (optional)
|Structured Streaming, consumer group
|Read from Korvet, write to Delta Lake

|S3
|Delta Lake format, ZSTD compression
|Cold storage tier
|===

== Prerequisites

* AWS CLI configured with appropriate permissions
* Terraform >= 1.3
* kubectl
* Helm 3.x

== Step-by-Step Deployment

=== Step 1: Deploy Infrastructure

[source,bash]
----
cd terraform
terraform init
terraform apply -var="region=us-west-2" -var="owner=your-name"
----

This creates:

* VPC with public/private subnets
* EKS cluster (6 nodes: 3 for Redis, 3 for workloads)
* S3 bucket for tiered storage
* IAM roles for S3 access (IRSA)

=== Step 2: Configure kubectl

[source,bash]
----
aws eks update-kubeconfig --name korvet-poc --region us-west-2
----

=== Step 3: Install EBS CSI Driver

[source,bash]
----
# Create IAM role for EBS CSI
./scripts/setup-ebs-csi.sh

# Verify driver is running
kubectl get pods -n kube-system | grep ebs
----

=== Step 4: Deploy Redis Enterprise

[source,bash]
----
# Install Redis Enterprise Operator
helm install redis-enterprise-operator \
  https://github.com/RedisLabs/redis-enterprise-helm/releases/download/redis-enterprise-operator-7.22.2-38/redis-enterprise-operator-7.22.2-38.tgz \
  -n redis --create-namespace

# Wait for operator
kubectl wait --for=condition=available deployment/redis-enterprise-operator -n redis --timeout=300s

# Deploy cluster and database
kubectl apply -f k8s/redis-enterprise-cluster.yaml
kubectl wait --for=condition=Running rec/rec -n redis --timeout=600s
kubectl apply -f k8s/redis-enterprise-database.yaml
----

=== Step 5: Deploy Korvet

[source,bash]
----
# Copy Redis password to korvet namespace
kubectl get secret redb-korvet-db -n redis -o json | \
  jq 'del(.metadata.resourceVersion, .metadata.uid, .metadata.creationTimestamp, .metadata.namespace, .metadata.ownerReferences)' | \
  kubectl apply -n korvet -f -

# Update k8s/korvet/serviceaccount.yaml with IAM role ARN from terraform output
# Update k8s/korvet/configmap.yaml with S3 bucket from terraform output

# Deploy Korvet
kubectl apply -f k8s/korvet/
kubectl wait --for=condition=available deployment/korvet -n korvet --timeout=120s
----

=== Step 6: Create Topic

[source,bash]
----
kubectl run kafka-cli --image confluentinc/cp-kafka:7.5.0 --restart=Never -n korvet -- sleep 3600

kubectl exec kafka-cli -n korvet -- kafka-topics \
  --bootstrap-server korvet.korvet.svc.cluster.local:9092 \
  --create --topic logs --partitions 4

# Configure tiered storage settings
kubectl exec kafka-cli -n korvet -- kafka-configs \
  --bootstrap-server korvet.korvet.svc.cluster.local:9092 \
  --entity-type topics --entity-name logs \
  --alter --add-config 'bucket.duration.ms=300000,retention.ms=1800000,remote.storage.enable=true'
----

IMPORTANT: These settings must be configured **before** data is written:

* `remote.storage.enable=true` - Enables archival to S3
* `bucket.duration.ms=300000` - Buckets seal every 5 minutes and get archived
* `retention.ms=1800000` - Data expires from Redis after 30 minutes (TTL = retention + bucket duration)

For production, use longer durations (e.g., `bucket.duration.ms=3600000` for 1 hour) to reduce S3 API calls.

=== Step 7: Deploy Logstash

[source,bash]
----
kubectl apply -f k8s/logstash/
kubectl wait --for=condition=available deployment/logstash -n korvet --timeout=120s
----

== Verification

=== Check Data Flow

[source,bash]
----
# Consume messages from Korvet
kubectl exec kafka-cli -n korvet -- kafka-console-consumer \
  --bootstrap-server korvet.korvet.svc.cluster.local:9092 \
  --topic logs --from-beginning --max-messages 5

# Check message offsets
kubectl exec kafka-cli -n korvet -- kafka-run-class kafka.tools.GetOffsetShell \
  --broker-list korvet.korvet.svc.cluster.local:9092 --topic logs --time -1
----

=== Check Redis (Hot Tier)

[source,bash]
----
REDIS_PASSWORD=$(kubectl get secret redb-korvet-db -n redis -o jsonpath='{.data.password}' | base64 -d)
kubectl exec rec-0 -n redis -- redis-cli -h korvet-db -p 11030 -a "$REDIS_PASSWORD" --no-auth-warning KEYS 'korvet:stream:logs:*'
----

=== Check S3 (Cold Tier)

[source,bash]
----
S3_BUCKET=$(cd terraform && terraform output -raw s3_bucket_name)
aws s3 ls s3://$S3_BUCKET/ --recursive --human-readable --summarize
----

=== Check Logstash Metrics

[source,bash]
----
kubectl port-forward deployment/logstash 9600:9600 -n korvet &
curl -s http://localhost:9600/_node/stats/pipelines | jq '.pipelines.main.events'
----

== Configuration

=== Tiered Storage Settings

[cols="1,2,3"]
|===
|Setting |Default |Description

|`remote.storage.enable`
|false
|**Must be set to `true`** to enable archival to S3

|`bucket.duration.ms`
|3600000 (1 hour)
|How often buckets are sealed and archived to S3

|`retention.ms`
|604800000 (7 days)
|TTL for data in Redis. Buckets expire after `retention.ms + bucket.duration.ms`
|===

IMPORTANT: TTL is set when data is written. If you change `retention.ms` after data exists,
old data keeps its original TTL. To apply new settings, recreate the topic.

NOTE: Size your Redis database to hold at least 2x the bucket duration worth of data
to allow for archival lag.

=== Scaling Logstash Throughput

Increase replicas or generator threads:

[source,yaml]
----
# k8s/logstash/deployment.yaml
spec:
  replicas: 2  # More replicas = more throughput

# k8s/logstash/configmap.yaml - increase threads
input {
  generator {
    threads => 8  # Default: 4
  }
}
----

== Troubleshooting

=== Redis Enterprise Not Starting

[source,bash]
----
kubectl describe rec/rec -n redis
kubectl logs -l name=redis-enterprise-operator -n redis
----

=== Korvet Cannot Connect to Redis

[source,bash]
----
kubectl logs -l app=korvet -n korvet | grep -i error
kubectl exec deployment/korvet -n korvet -- env | grep KORVET
----

=== S3 Permission Issues

[source,bash]
----
kubectl get sa korvet -n korvet -o yaml | grep eks.amazonaws.com
kubectl exec deployment/korvet -n korvet -- aws sts get-caller-identity
----

=== No Data in S3

Check that:

1. Topic has `bucket.duration.ms` set (default 1 hour may be too long for testing)
2. Korvet is using `early-access` image (tiered storage fully supported)
3. Enough time has passed for bucket to seal
